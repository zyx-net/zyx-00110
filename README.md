# 设备巡检停复机登记系统

## 项目概述

本系统是一个基于Python Tkinter的设备巡检停复机登记桌面工具，用于管理设备的异常报告、停机审批、维修登记和复机审批流程。

## 功能特性

- 设备管理：添加、删除设备，查看设备状态
- 状态流转：正常 → 待停机审批 → 已停机 → 维修中 → 待复机审批 → 已复机 → 复发观察
- 维修记录：记录维修内容、维修人员和维修时间
- 审批记录：记录停机/复机审批意见和审批人
- **主管权限控制**：复机审批需要主管身份，普通人员无法批准复机
- 阈值配置：配置监控阈值
- 数据导出：**同时导出JSON和CSV格式**，包含设备状态、维修记录、审批记录
- **数据导入与回滚**：从历史导出文件导入数据，支持预检、权限控制、自动备份和撤销
- 数据持久化：所有数据本地存储，关闭再打开后数据不丢失

## 状态流转图

```
正常 ──(报告异常)──→ 待停机审批 ──(申请停机)──→ 已停机
   ↑                                                   │
   │                                                   ↓
   └──(恢复正常)── 复发观察 ←──(复发观察)── 已复机 ←──(批准复机)── 待复机审批
                                                          ↑
                                                          │
                                                   (完成维修)── 维修中
```

## 实操步骤

### 主流程验证

#### 1. 添加设备
- 在"设备管理"标签页中，输入设备ID（如 `DEV001`）和设备名称（如 `水泵一号`）
- 点击"添加设备"按钮

#### 2. 巡检发现异常
- 选中刚添加的设备
- 点击"报告异常"按钮
- 在弹出对话框中输入异常描述（如 `水泵异响，振动超标`）
- 点击确定，设备状态变为"待停机审批"

#### 3. 申请停机
- 选中"待停机审批"状态的设备
- 点击"申请停机"按钮
- 在弹出对话框中输入停机原因（如 `需检修轴承`）
- 点击确定，设备状态变为"已停机"

#### 4. 开始维修
- 选中"已停机"状态的设备
- 点击"开始维修"按钮
- 设备状态变为"维修中"

#### 5. 登记维修记录
- 选中"维修中"状态的设备
- 点击"完成维修"按钮
- 在弹出对话框中输入维修内容（如 `更换轴承，润滑处理`）
- 输入维修人员（如 `张三`）
- 点击确定，设备状态变为"待复机审批"

#### 6. 主管批准复机
- **重要**：只有主管（李四、王五、赵六、admin）才能批准复机
- 选中"待复机审批"状态的设备
- 点击"批准复机"按钮
- 在弹出对话框中输入主管姓名（如 `李四`）
- 点击确定，如果身份验证通过，设备状态变为"已复机"
- 如果输入普通人员姓名（如 `张三`），系统会拒绝并提示权限不足

#### 7. 导出记录
- 切换到"系统配置"标签页
- 点击"浏览"按钮选择导出目录（或使用默认目录）
- 点击"导出记录"按钮
- **会同时生成JSON和CSV两个文件**

#### 8. 重启验证
- 关闭应用后重新打开
- 验证设备状态、维修记录、审批记录都完整保留

### 数据导入与回滚验证

#### 1. 准备导出文件
- 在"系统配置"标签页点击"导出记录"，生成 JSON 和 CSV 导出文件
- 记录当前设备数量和状态

#### 2. 验证身份
- 切换到"数据导入"标签页
- 在"操作人"输入框输入姓名（如 `admin`）
- 点击"验证身份"按钮
- 如果是主管（李四、王五、赵六、admin），显示"已验证 - 主管 (可导入)"
- 如果是普通人员，显示"已验证 - 普通人员 (仅预览)"

#### 3. 选择导入文件
- 点击"浏览"按钮，选择之前导出的 JSON 或 CSV 文件
- 支持两种格式的导入

#### 4. 预检导入
- 点击"预检导入"按钮
- 系统会分析文件内容，显示预检结果：
  - **新增**：文件中有但系统中不存在的新记录
  - **覆盖**：文件中有且系统中已存在的记录（将覆盖）
  - **冲突**：存在重复或冲突的记录
  - **无效**：数据不合法的记录（如状态不合法、审批人无权限等）
- 普通人员只能预览，无法执行导入

#### 5. 执行导入（仅主管）
- 主管确认预检结果后，点击"执行导入"按钮
- 系统会自动备份当前数据到 `data/backups/` 目录
- 导入完成后，设备列表和历史日志会自动刷新
- 撤销信息会显示在界面上

#### 6. 撤销导入（仅主管）
- 导入后，如果需要撤销，点击"撤销上次导入"按钮
- 系统会从备份恢复数据
- 撤销后，撤销状态会被清除

#### 7. 重启后状态保持
- 关闭应用后重新打开
- 切换到"数据导入"标签页
- 验证：
  - 撤销状态仍然可用（如果有未撤销的导入）
  - 导入日志仍然存在
  - 配置保持不变

### 导入错误处理验证

#### 1. 设备编号重复
- 尝试导入包含已存在设备ID的文件
- 预检结果会显示在"覆盖"标签页

#### 2. 状态不合法
- 修改导出文件，将设备状态改为无效值（如"无效状态"）
- 预检时会在"无效"标签页显示错误信息："状态不合法: '无效状态'"

#### 3. 审批人无权限
- 修改导出文件，将复机审批的审批人改为普通人员
- 预检时会在"无效"标签页显示错误信息："审批人'xxx'无复机审批权限"

#### 4. CSV/JSON 字段缺失
- 修改导出文件，删除必填字段（如设备ID或名称）
- 预检时会在"无效"标签页显示错误信息："设备ID缺失"或"设备名称缺失"

### 失败链路验证

#### 1. 没有维修记录却申请复机
- 添加一个新设备，走完流程到"待复机审批"
- 直接修改数据文件，清空维修记录（修改 `data/repair_records.json` 为空数组）
- 重新打开应用，选中该设备，点击"批准复机"
- 输入主管姓名
- 系统提示："没有维修记录，不能批准复机"

#### 2. 已停机设备再次重复停机
- 选中"已停机"状态的设备
- 点击"申请停机"按钮
- 系统提示："设备已停机，不能重复停机"

#### 3. 导出目录改成非法路径
- 切换到"系统配置"标签页
- 在"导出目录"输入框中输入不存在的路径（如 `C:\nonexistent\dir`）
- 点击"保存配置"按钮
- 系统提示："导出目录不存在"，原有设置不会被冲掉
- 重新打开应用，验证配置仍然是原来的路径

#### 4. 普通人员无法批准复机
- 选中"待复机审批"状态的设备
- 点击"批准复机"按钮
- 输入普通人员姓名（如 `普通员工`）
- 系统提示："权限不足，'普通员工'不是主管，无权批准复机"
- 只有输入以下主管之一才能成功：李四、王五、赵六、admin

#### 5. 普通人员无法执行导入
- 切换到"数据导入"标签页
- 输入普通人员姓名并验证身份
- 预检导入可以正常执行
- 点击"执行导入"按钮
- 系统提示："权限不足：只有主管才能执行导入操作"

1. **NameError崩溃**：保存导出目录或导出记录时会因缺少 `os` 和 `json` 模块导入而崩溃

3. **仅JSON导出**：只能导出JSON格式，没有CSV格式
4. **配置可能丢失**：非法路径保存时可能冲掉原有配置
5. **Windows编码错误**：`python test_regression.py` 在默认Windows终端会因Unicode符号触发 UnicodeEncodeError
6. **无导入功能**：无法从历史导出文件恢复数据

1. ✅ **稳定运行**：所有文件操作正常，不会抛出NameError
2. ✅ **权限验证**：复机审批需要主管身份，普通人员无法通过
3. ✅ **双格式导出**：同时生成JSON和CSV两个文件
4. ✅ **配置保护**：非法路径保存失败时，原配置完整保留
5. ✅ **主管列表可见**：复机审批时显示可审批的主管列表
6. ✅ **终端兼容**：测试脚本使用ASCII字符输出，普通Windows终端无需设置 PYTHONIOENCODING 即可运行
7. ✅ **数据导入**：支持从 JSON 和 CSV 文件导入设备、维修和审批记录
8. ✅ **预检功能**：导入前预览新增、覆盖、冲突和无效记录
9. ✅ **权限控制**：普通人员只能预览，主管才能执行导入
10. ✅ **自动备份**：导入前自动备份当前数据
11. ✅ **撤销功能**：支持撤销最近一次导入
12. ✅ **状态持久化**：重启后导入日志和撤销状态仍然可用

## 数据持久化

所有数据存储在 `data/` 目录下的JSON文件中：

- `devices.json`：设备信息
- `repair_records.json`：维修记录
- `approval_records.json`：审批记录
- `config.json`：系统配置（导出目录、阈值等）
- `import_logs.json`：导入操作日志
- `rollback_state.json`：撤销状态（保存最近一次可撤销的导入信息）

关闭应用后再打开，所有数据将自动加载恢复。

## 注意事项

1. 请确保运行目录有读写权限
2. 导出目录必须是已存在的有效目录
3. 设备ID不能重复
4. 状态流转必须按照规定的顺序进行
5. **复机审批必须使用主管身份**，普通人员无法通过权限验证
6. **数据导入必须使用主管身份**，普通人员只能预览
7. 每次导入都会自动备份，但只保留最近一次可撤销的导入
8. 撤销操作只能执行一次，撤销后无法再次撤销
